# Deploying ExamPro — Vercel + Supabase + Render

This guide takes the app from `localhost:4000` to production on the PRD §8 stack.
Two tracks — use **Track A to be live today**, **Track B for the full PRD architecture**.

| | Track A (fast) | Track B (PRD §8) |
|---|---|---|
| Frontend | Served by Render (same process) | **Vercel** |
| Backend | **Render** web service | **Render** web service (API-only, CORS on) |
| Database | SQLite on a Render **persistent disk** | **Supabase Postgres** |
| Files | Same disk | **Supabase Storage** |
| Live dashboard | SSE via the API | SSE now / **Supabase Realtime** optional |
| Code changes | None | Postgres adapter (see §B.6) |

## 0. One-time prep (both tracks)

1. Put the code on GitHub:
   ```bash
   cd exampro
   git init && git add -A
   git commit -m "ExamPro — digital examination platform"
   git remote add origin git@github.com:YOU/exampro.git
   git push -u origin main
   ```
2. **Never commit `.env`** — it contains your Gemini key. It's already excluded via
   `.gitignore` (verify: `git status` must not list `exampro/.env`).
3. ⚠️ **Rotate your Gemini key.** It was shared over chat — treat it as exposed:
   [aistudio.google.com/apikey](https://aistudio.google.com/apikey) → delete the old key →
   create a new one. You'll paste the new one into Render (step A.3 / B.3), never into git.

## Track A — all-in-one on Render (live in ~15 min)

The Node server already serves the built frontend + API together; nothing to change.

### A.1 Create the service
Render Dashboard → **New → Blueprint** → connect your repo. Render reads
[`render.yaml`](./render.yaml) and provisions everything (service, disk, secrets prompts).
Prefer manual? **New → Web Service** with:

| Setting | Value |
|---|---|
| Runtime | Node |
| Build Command | `npm run setup && npm run build` |
| Start Command | `node server/src/index.js` |
| Health Check Path | `/api/health` |
| Disk | 1 GB mounted at `/opt/render/project/src/server/data` |

### A.2 Seed the production database
In Render → your service → **Shell**:
```bash
node server/src/db/seed.js
```
(The disk keeps it across deploys. Re-run only to reset demo data.)

### A.3 Environment variables (Render → Environment)
| Key | Value |
|---|---|
| `GEMINI_API_KEY` | your (rotated) key — enables AI Studio |
| `GEMINI_MODEL` | `gemini-2.5-flash` |
| `SESSION_SECRET` | auto-generated by the blueprint |
| `DATABASE_FILE` | `/opt/render/project/src/server/data/exampro.db` |
| `NODE_VERSION` | `20` |

`CLIENT_ORIGIN` / `COOKIE_SECURE` / `DATABASE_URL` stay **empty** on Track A.

### A.4 Verify
- `https://YOUR-SERVICE.onrender.com/api/health` → `{"ok":true,...}`
- Teacher login → dashboard; `/exam` → student login (`STU-1001` / `CARD-7291`).
- **Cold starts:** the free plan sleeps after 15 min idle — first hit takes ~30–60 s
  (the student lobby shows a loading state; fine for demos). Starter plan ($7) removes this.

Production tips on Track A: turn on **Render disk backups** (snapshots), set
`PLAN=starter` for exams with >100 concurrent students, and add a custom domain
(Render → Settings → Domains — free TLS).

## Track B — PRD stack: Supabase + Render + Vercel

### B.1 Supabase project (database + storage + cron + realtime)
1. [supabase.com](https://supabase.com) → **New project** → choose region close to your
   users → set a strong DB password → wait ~2 min.
2. **SQL Editor → New query** — run these scripts **in order** (from this repo's
   `supabase/` folder):
   | # | File | What it does |
   |---|---|---|
   | 1 | `schema.pg.sql` | All 17 tables, Postgres dialect |
   | 2 | `cron.sql` | `pg_cron` every-minute cutoff marking (PRD §4.1) + nightly session purge |
   | 3 | `realtime.sql` | Publishes `attempts`/`violations`/`answers` over Realtime (PRD §6.1) |
   | 4 | `storage.sql` | Private `exampro-notes` bucket (PRD §4.8/4.9) |
   | 5 | `rls.sql` | Row Level Security scoping direct connections to custom JWT claims (PRD §8) — apply with the §B.6 cutover |
3. Collect credentials (**Project Settings → API / Database**):
   - `Connection string → Session pooler` (use this from Render — IPv4-safe),
     e.g. `postgresql://postgres.xxxx:PASS@aws-0-eu-central-1.pooler.supabase.com:5432/postgres`
   - `service_role` key (server-side only — bypasses RLS) and `anon` key (browser Realtime).

### B.2 Render backend (API-only)
Same service definition as Track A (repo's `render.yaml` works), different env:

| Key | Value |
|---|---|
| `DATABASE_URL` | Supabase session-pooler string (§B.1) |
| `DATABASE_FILE` | *(delete — PG takes precedence)* |
| `CLIENT_ORIGIN` | `https://YOUR-APP.vercel.app` (required — enables CORS) |
| `COOKIE_SECURE` | `true` (cross-site cookies) |
| `GEMINI_API_KEY` | rotated key |
| `SESSION_SECRET` | strong random |
| `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` | only used after the Storage cutover (§B.6) |

The Render service keeps: all REST routes, Gemini calls + async grading queue
(PRD §8's "slow AI work off Vercel"), custom roll-number+access-code auth/RLS-claim
minting, the 15 s grading sweep (pg_cron marks due rows; the sweep grades them),
and the SSE monitor stream.

Health: `https://YOUR-API.onrender.com/api/health` → `{"ok":true}`.

### B.3 Vercel frontend
1. [vercel.com](https://vercel.com) → **Add New → Project** → import the repo.
2. **Root Directory → `web`** (Vercel auto-detects Vite; the included
   [`web/vercel.json`](./web/vercel.json) pins the build + SPA fallback).
3. **Pick ONE way to reach the API:**

   **Option 1 — proxy through Vercel (recommended).** Edit `web/vercel.json` and
   replace `YOUR-RENDER-SERVICE.onrender.com` with your API host. Browser → Vercel
   stays first-party → cookies just work, no third-party-cookie pitfalls; SSE streams
   through the rewrite. Leave `VITE_API_BASE` **unset**.

   **Option 2 — direct cross-origin.** Delete the `/api` rewrite from `web/vercel.json`,
   then set Vercel env `VITE_API_BASE=https://YOUR-API.onrender.com`. The SPA then calls
   Render directly (built-in support: `Authorization: Bearer` fallback + credentials).
   Requires `CLIENT_ORIGIN` + `COOKIE_SECURE=true` on Render (§B.2) and works even where
   browsers block third-party cookies.

4. Deploy → your app is at `https://YOUR-APP.vercel.app`.

### B.4 Wire & verify (10-point checklist)
1. `/api/health` OK on Render.
2. Teacher login on the Vercel URL → dashboard loads (cookies or Bearer both accepted).
3. Create a course + one MCQ exam, add a roster row, note the access code.
4. Student login at `/exam` → lobby shows fixed end time; start exam.
5. Timer ticks from `remaining_ms` (server value) — refresh-proof.
6. Tab-switch once → strike 1 banner → teacher **Live Monitoring** grid updates
   (SSE/Realtime within ~2 s).
7. Submit → result shows MCQ auto-score immediately (provisional if essays exist).
8. AI Studio → upload notes → **Generate** → items appear in review queue (Gemini path OK).
9. Teacher confirms grade → results release → student sees final + PASS/FAIL.
10. After the exam end passes, `select status from attempts;` in Supabase shows
    `auto_submitted` even with Render asleep (pg_cron working).

### B.5 Production hardening
- Restrict `CLIENT_ORIGIN` to your exact Vercel domain (no wildcards) — already enforced server-side.
- Supabase: verify RLS enabled (`rls.sql`), keep `service_role` server-side only, set up **scheduled backups** (Supabase does daily on paid plans; take manual dumps on free).
- Render: enable auto-deploy from `main`; add a **Preview** service for staging.
- Vercel: enable **Preview Deployments**; add your custom domain.
- Consider Cloudflare or Render's built-in rate limiting in front of the student login route (it's the public attack surface).

### B.6 The one engineering step left → database/storage/realtime adapter
The app's data layer is SQLite (`better-sqlite3`, synchronous). The schema is fully
converted in `supabase/schema.pg.sql` and the runtime already accepts `DATABASE_URL`,
but the final cutover needs a small Postgres adapter behind the existing `q.get/q.all/q.run`
API (async, `pg` driver, `$1` params, `RETURNING id` instead of `lastInsertRowid`), plus:
- notes upload → Supabase Storage (`stored_path` becomes an object key),
- (optional) SSE monitor stream → Supabase Realtime channel on the Vercel frontend,
- JWT claim minting for the RLS policies in `rls.sql` (PRD §8).
